# ContextBridge Kernel — Storage Engine Specification (RFC 001)

## Philosophy

The storage layer is strictly decoupled from both the ABI (`memory.py`) and the retrieval mechanisms.
It must allow the **Memory Normalizer** to write canonical nodes independently of the **Embedding Engine** generating search indexes.

---

## 1. Core Graph Storage

### `memory_objects` (The Source of Truth)

The fundamental node of the Knowledge Graph. It contains zero AI-specific logic.

* `id` (UUID, PK)
* `project_id` (UUID, FK -> projects)
* `type` (Enum: conversation, decision, bug, architecture, checkpoint, etc.)
* `title` (Varchar 255)
* `body` (Text)
* `importance` (Float, 0.0-1.0)
* `confidence` (Float, 0.0-1.0)
* `freshness` (Float, 0.0-1.0)
* `owner` (Varchar 255)
* `status` (Enum: active, superseded, rejected)
* `created_at` (Timestamp)
* `updated_at` (Timestamp)

### `relationships` (The Edges)

Allows hierarchical tree traversal and graph search *before* ever hitting an embedding layer.

* `id` (UUID, PK)
* `source_id` (UUID, FK -> memory_objects)
* `target_id` (UUID, FK -> memory_objects)
* `relationship_type` (Enum: depends_on, blocks, decides, mentions, implements)
* `weight` (Float, 0.0-1.0)
* `confidence` (Float, 0.0-1.0)

---

## 2. Pluggable Indexing

### `embeddings` (The Search Indexes)

Strictly separated from memory. A single memory object can have multiple embeddings generated by different models (e.g., Voyage, OpenAI, Jina).

* `id` (UUID, PK)
* `memory_id` (UUID, FK -> memory_objects)
* `provider` (Varchar, e.g., 'openai')
* `model` (Varchar, e.g., 'text-embedding-3-small')
* `dimensions` (Integer, e.g., 1536)
* `vector` (Vector column via pgvector)

---

## 3. Provenance & Evolution

### `memory_versions` (The Evolution Log)

Enables diffing, rollback, and historical tracking of architectural decisions or project goals over time.

* `id` (UUID, PK)
* `memory_id` (UUID, FK -> memory_objects)
* `version_number` (Integer)
* `diff` (JSONB)
* `author` (Varchar)
* `created_at` (Timestamp)

### `evidence` (The Explanation Layer)

Separates the "Memory" (a synthesized decision) from the "Evidence" (the raw sources that support it).

* `id` (UUID, PK)
* `memory_id` (UUID, FK -> memory_objects)
* `source_id` (UUID, FK -> sources)
* `reasoning` (Text)

---

## 4. Provider & Capability Registry

### `providers`

Registers authorized integration agents.

* `id` (UUID, PK)
* `name` (Varchar, e.g., 'genesis', 'git', 'chrome_extension')
* `version` (Varchar)

### `provider_capabilities`

A mapping that allows the Memory Planner to request capabilities rather than hardcoding providers.

* `provider_id` (UUID, FK -> providers)
* `capability` (Enum: read, write, watch, search, stream, diff, checkpoint)

---

## 5. Summary of Retrieval Flow (V2)

1. **Intent Detection:** "What is the user trying to do?"
2. **Planner:** Filters via `memory_objects.type` and `provider_capabilities`.
3. **Graph Traversal:** Navigates `relationships` to pull supporting evidence.
4. **Embedding Ranking:** (Only used to rank the already-filtered subgraph).
5. **Assembler & Formatter:** Compiles for the specific LLM API.
